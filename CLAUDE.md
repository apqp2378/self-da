# CLAUDE.md — Career-Ops Module A 작업 지침

## 배치
- 이 파일은 **self-da 레포 최상위 루트**(`self-da/CLAUDE.md`)에 둔다. 그 폴더에서 세션을 열면 자동 로드된다.
- Module A(채용공고 BERTopic 토픽 분류 + 스킬 빈도)는 self-da 안 **`da_toolkit/module_a/`**에 둔다.
- Module A는 self-da **Week 1(진단·정의)의 JD Parser → Gap Matcher를 심화**하는 모듈이다. Week 2 NL→SQL 해커톤은 그대로 둔다.

## 4원칙 (Karpathy)

### 원칙 1: 코딩 전에 먼저 생각하기
추측하지 말 것. 여러 해석 가능하면 모두 제시하고 선택지를 줄 것.
구현 전에 입력·출력·예외 케이스를 먼저 정의한다.

### 원칙 2: 최소한의 코드
요청한 기능 외 추가하지 말 것. 100줄로 해결되면 1000줄 짜지 말 것.
유틸 함수, 추상 클래스, 데코레이터는 명시적으로 요청받기 전에 만들지 않는다.

### 원칙 3: 수술적 수정
내가 요청한 부분만 수정. 관련 없는 코드·주석 건드리지 말 것.
수정 후 "내가 요청한 것만 변경했습니다"를 명시.

### 원칙 4: 성공 기준 기반 실행
"버그 고쳐줘" (X) → "버그 재현 테스트 작성 후 통과시켜줘" (O)
"완료했습니다" 대신 검증 결과를 보고할 것.

## Career-Ops 추가 원칙

### 원칙 5: 분석 단위 분리
JD 1건, 토픽 1개, 스킬 1건 단위를 절대 혼동하지 않는다.
모든 DataFrame, 함수의 입력·출력에 단위를 docstring으로 명시한다.

예:
```python
def aggregate_skills_by_topic(jd_skill_df: pd.DataFrame) -> pd.DataFrame:
    """
    입력 단위: JD 1건 × 스킬 1건 = 1행 (long format)
    출력 단위: 토픽 1개 × 스킬 1건 = 1행 (long format)
    """
```

### 원칙 6: LLM 사용처 제한
파싱·매칭·집계 같은 결정론적 작업에는 Claude API를 호출하지 않는다.
LLM은 다음에만 사용한다:
- 토픽 라벨링 검수 (BERTopic이 만든 키워드 묶음 → 사람이 읽을 수 있는 토픽명)
- JD 안의 모호한 표현 정규화 ("우대: 대시보드 경험" → "BI 도구 사용 경험" 같은 매핑)

### 원칙 7: 운영 보조 지표 위치
모듈 A의 모든 산출물은 "본인이 채용공고 결을 좁히는 보조 자료"이지 분류 정답이 아니다.
- 토픽 외 unknown 라벨을 허용한다.
- 스킬 빈도는 절대 점수가 아니라 상대 비교 자료다.
- 모든 시각화/표에 "본 자료는 운영 보조 지표로 활용한다"를 footer로 명시한다.

### 원칙 8: 외부 도구 권한 제한
MCP 커넥터(Notion/GitHub 등)를 사용할 때 다음을 지킨다.
- 도구 권한 기본값은 **"승인 필요"**. "항상 허용"은 토큰 비용·보안 양면 위험이 있어 명시적 사유 없이 사용하지 않는다.
- **워크스페이스/레포 단위로 권한을 분리**한다. Notion 전체 워크스페이스 권한, GitHub 모든 레포 권한을 한 번에 부여하지 않는다.
- **신뢰하지 않는 MCP URL을 추가하지 않는다**. 디렉토리에 없는 커스텀 커넥터는 출처를 확인한 뒤에만 사용한다.
- **민감 데이터를 MCP 접근 가능 위치에 평문으로 두지 않는다** (GitHub PAT, API 키, 개인정보 등).
- 매월 1회 `~/.claude.json`(또는 `~/Library/Application Support/Claude/claude_desktop_config.json`)에 비정상 MCP 서버 URL이 추가되지 않았는지 점검한다.
